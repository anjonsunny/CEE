# Ground Truth Generation Protocol

This file documents how Claude (in-terminal) generates candidate causal graphs
that the CEE+ Ground Truth tab loads and verifies.

## Role

I (Claude) am an **independent LLM reasoner**, not external ground truth.
The pipeline is:

1. I generate candidate caption + causal graph for each image.
2. User verifies (accepts / edits / rejects) each candidate via the Ground Truth tab.
3. The verified subset becomes the reference for Test 1 (Ground Truth Comparison).

Methodological framing for the paper: "candidate reference generated by a
different LLM (Claude), validated by author spot-check on N=X scenes." Not
ground truth in the strict sense — a cross-model reference.

## Inputs I receive

- Path to a folder containing images (and only images — no captions needed).
- The folder may have category subfolders (e.g. `fire/palisade/`, `flood/helene/`).
  In that case images with identical basenames across subfolders collide. I
  handle this with the **namespaced filename convention** (see below).

## Namespaced filename convention

To avoid basename collisions across category subfolders, candidate GT files
use a disambiguated stem:

    <flat_category>__<basename>.gt.json

where `<flat_category>` is the path of the image relative to the root folder,
with `/` replaced by `_`. Images in a flat folder (no subfolder) keep their
plain basename.

Examples (with root = `experiments/exp2/images/`):

    fire/palisade/1.jpg   →  fire_palisade__1.jpg.gt.json
    flood/helene/1.jpg    →  flood_helene__1.jpg.gt.json
    1.jpg (at root)       →  1.jpg.gt.json

The GT JSON's `image_filename` field stores the same namespaced name. CEE+'s
batch worker (`namespaced_image_name`) produces the matching key, so Test 1's
verified-GT lookup joins cleanly on this field.

## What I produce for each image

A `<image_basename>.gt.json` file next to a copy of the image in
`exports/ground_truth/candidates/`. JSON shape:

```json
{
  "image_filename": "<namespaced_filename>",
  "caption": "<1-2 sentence neutral description of the scene>",
  "source": "claude-code (in-terminal generation)",
  "annotator_notes": "",
  "nodes": [
    {
      "id": "<label>_<N>",
      "label": "<singular noun: house|man|woman|child|car|tree|dog|...>",
      "state": "<adjective from state vocabulary>",
      "hazardous": <true|false>,
      "inferred": false
    }
  ],
  "edges": [
    {
      "source": "<node id>",
      "target": "<node id>",
      "effect": "<one of 8 effect labels>",
      "via_state": "<hazard-bearing state of source>"
    }
  ]
}
```

## Schema constraints I follow

These match CEE+'s Prompt 2 vocabularies verbatim so comparisons are clean.

### Object IDs
`<label>_<N>`, 1-indexed per label. Use the most specific applicable label
(man/woman/child > person, dog/cat > animal).

### State vocabulary

Hazard-bearing: burning, burnt, collapsed, collapsing, fallen, crushed, flooded,
leaking, bleeding, injured, approaching, charging, aiming, coiled, rabid,
armed, fleeing, striking, rising, spreading, billowing, seeping, escalating

Note on present-vs-past participle: most active states have a settled counterpart
for "the threat is no longer in motion, but the entity that was hit is now a
hazard itself":
  burning → burnt (charred shell, contact hazard)
  collapsing → collapsed (structurally fallen, contact hazard)
  rising → flooded (water has come and settled on entity)
  falling → fallen (tree is down)
Use the past-participle for *settled* aftermath when the entity itself is still
dangerous to touch/approach. Use the present-participle only when motion or
imminent collapse is visible in-frame.

## Distinguishing burnt vs collapsed (state describes the entity's
   current visual condition, not the cause of damage)

  burnt:      entity is still recognizable as a structure / object — its
              outline, walls, or frame are visible — but visibly fire-
              consumed (charred, gutted, hollowed). Threats: sharp metal,
              hot spots, toxic residue, asbestos, unstable interior on
              contact.

  collapsed:  entity is no longer recognizable as a coherent structure —
              rubble pile, walls down, roof gone, just material on the
              ground. Threats: void hazards, unstable mound, shifting
              debris on contact.

  collapsing: in-motion or visibly imminent collapse (cantilevered parts,
              dangling material, mid-fall). Threats: falling material on
              anyone nearby.

The cause (fire / earthquake / storm / explosion) is conveyed by adjacent
entities and scene context, not by the state. A flat fire-rubble lot and a
flat earthquake-rubble lot are both `collapsed`, since they pose the same
visual contact threats.

Normal: intact, standing, upright, whole, dry, sealed, uninjured, healthy,
stationary, resting, disengaged, relaxed, unarmed, stable, contained,
dissipating, steady

An object whose state is hazard-bearing has `hazardous: true`. Period.
State alone is sufficient — the entity does NOT need to be the source of
any edge to be marked hazardous. A burnt-out shell with no one around is
still hazardous; a flooded car with only an incoming `water_1 → car_1`
edge is still hazardous. The hazard flag describes the entity's intrinsic
condition, not its graph role.

Edge invariants on hazardous nodes:
  - A hazardous node with no plausible target in scene → add self-loop
    `node_id → node_id worsens, via_state=<current state>`. This captures
    "intrinsically deteriorating" and avoids a fully disconnected node.
  - A hazardous node with only incoming edges (pure casualty — e.g.,
    flooded car hit by water_1) is allowed. No outgoing edge required.
  - A hazardous node with outgoing edges to other entities is the standard
    pattern (e.g., fire threatening firefighter).

**Fluid / gaseous hazards convention (matches Prompt 1 and Prompt 2).** For
water, smoke, gas, dust, and free-burning fire-as-substance, emit the fluid
itself as its own entity with an *active* state. Examples:
  - Flood scene: `water_1` (state `rising` or `spreading`) is the hazard;
    a flooded car is the affected entity, NOT the threat source.
  - Smoke scene: `smoke_1` (state `billowing`) is the hazard; a smoke-filled
    building is affected, not the source.
  - Gas leak: `gas_1` (state `leaking` or `seeping`) is the hazard.

This convention disambiguates a schema-level question: many "hazard-bearing"
states (e.g., `flooded`, `collapsed`, `crushed`) describe entities that have
been HARMED rather than entities actively CAUSING harm. The fluid-as-entity
convention picks the causally accurate framing.

**Suppression / response elements are NOT entities.** The graph is a *threat*
graph — it exists to justify recommendations. Things being done *in response
to* a threat don't belong in the node set, because the 8 effect labels can
only express harm pathways, not mitigation. Specifically:

  - **Suppression fluids**: water from a fire hose, aerial water/retardant
    drops, foam, dry chemical. NOT modeled as entities. The action is
    captured in the caption and (if needed) `annotator_notes`.
  - **Suppression tools**: hoses, ladders, axes, pulaski tools, fire-breaks,
    backfires. NOT entities.
  - **Response vehicles**: fire engines, ambulances, helicopters performing
    suppression, evacuation buses. Generally NOT entities, unless one is
    itself at risk from a separate threat (then it appears as an affected
    entity but never as a threat source).
  - **PPE worn by responders**: never an entity.

Contrast with FLOOD water, BURNING-BUILDING smoke, FLOWING-LAVA, etc., which
ARE entities because they are sources of harm. The same physical substance
(water) is an entity in a flood and not an entity in a fire hose — its role
in the *threat* causal structure determines whether it appears.

Rule of thumb: if removing the candidate node leaves the recommendation
story intact, it doesn't belong in the graph.

### Effect vocabulary

- may_spread_to    — hazard propagates via physical contiguity
- may_harm         — threat injures target without target becoming a hazard
- blocks_access_to — physical obstruction
- isolates         — cuts off escape or resources
- exposes          — protective barrier removed/breached
- increases_risk_to — enabling factor; converts normal entity to latent threat
- worsens          — escalates a hazard on the SAME entity
- threatens        — last resort (only when no other applies)

### Edge constraints

- `via_state` must equal the source node's state and be hazard-bearing
- self-loops (source == target) only allowed with effect `worsens`

### Intrinsic-hazard self-loop convention

When an entity is in a hazard-bearing state but the scene contains no targets
(no people, no adjacent structures, no other entities the hazard can plausibly
threaten), use a self-loop with effect `worsens` on the entity itself:

  - `house_1` burning, alone in scene → edge `house_1 → house_1 worsens, via_state=burning`
  - `bus_1` fallen, no one nearby → edge `bus_1 → bus_1 worsens, via_state=fallen` (if collapse is ongoing)

This expresses "the hazard intrinsically degrades the entity that hosts it"
and satisfies the schema invariant *every hazardous node has at least one
outgoing edge* without inventing fake targets. Use sparingly — only when no
in-frame target plausibly exists. If even a presumed (inferred) occupant is
plausible and inferred mode is enabled, prefer that.

When the lone hazard would produce a *byproduct* entity (smoke from a burning
house, dust from a collapsing wall) and that byproduct has no targets either,
**drop the byproduct entity** rather than self-looping it. Smoke and dust are
not self-escalating; the host entity's self-loop captures the dynamics.

## How graphs are compared (matching semantics)

When Test 1 compares a candidate GT graph against the VLM's Graph A or Graph B,
the matcher operates at three levels of abstraction. **GT files have no
bounding boxes**, so there is no way to anchor `smoke_1` in GT to `smoke_1` in
Graph A by visual identity. The three matchers handle this gap differently:

**Strict** (`compare_graphs`) — verbatim 4-tuples `(source_id, target_id, effect,
via_state)`. Two edges match only if all four fields are identical. This is
deliberately brittle: it requires both *causal agreement* AND *ID convention
agreement*. If the VLM enumerates IDs in a different order than Claude
did, strict scores drop even on causally-identical graphs. Expect strict
to be the lowest of the three numbers.

**Soft** (`compare_graphs_soft`) — strips IDs and compares **label/state/effect
multisets**. IDs collapse to label classes (man / woman / child / firefighter →
`person`) via `LABEL_HIERARCHY`. States canonicalize via `STATE_SYNONYMS`
(charred → burnt, ablaze → burning). Effects collapse via `EFFECT_CLOSE_PAIRS`
(may_harm ≈ threatens; blocks_access_to ≈ isolates). Multisets preserve count:
"GT had 4 (fire, may_harm, person) edges; Graph A had 4 of the same → exact
match." This is the load-bearing number for the paper.

**Topological** (`compare_graphs_topological`) — like soft but ignores state
entirely. Just label + effect + label. Answers "did the model recover the
structural skeleton at all?" Useful when the model and Claude disagree on
state vocabulary (e.g., `burnt` vs `collapsed` on a fire-damaged shell) but
agree on what threatens what.

### Why we don't have bboxes

By design. Adding bboxes would mean: (a) Claude needs reliable visual grounding
during GT generation (currently we use vision but emit no boxes); (b) the
VLM's Graph A would need to emit boxes for matching, which Qwen doesn't do
reliably in our schema; (c) the research question shifts from "does the VLM
reason causally" to "does the VLM ground entities visually" — different paper.

For our purposes, the **soft multiset match** treats causal structure as the
unit of comparison, not entity identity. This is defensible because the
recommendations downstream are also type-level ("evacuate the person threatened
by the fire" doesn't change meaning if person_1 vs person_2 swap).

### Known limitation

If a scene has two distinct entities of the *same* label class and the GT
pairs them asymmetrically (e.g., `fire_1` threatens `firefighter_1` but
deliberately NOT `firefighter_2`), the soft multiset matcher can't tell them
apart — it sees `(fire, may_harm, person)` × 1 either way. Topological loses
even more. This is the cost of forgoing bbox anchoring; report soft + topo
alongside strict in the paper so readers can see the gap.

## How I reason

For each image:

1. **Caption** — one neutral sentence describing what I see.
2. **Detected objects** — enumerate every visible entity (or strongly implied,
   e.g., smoke from a fire). Assign object_ids. Pick the most appropriate state
   from the vocabulary.
3. **Threats** — mark `hazardous: true` for entities in hazard-bearing states
   that have outgoing edges (i.e., they actually reach something else).
4. **Edges** — for each hazardous entity, enumerate every causal pathway it
   has to other entities in the scene. Include:
   - Direct harm (e.g., burning house → person)
   - Cascade (e.g., burning house → adjacent house via may_spread_to)
   - Latent conversion (e.g., burning house → sealed gas tank via increases_risk_to)
   - Exposure / isolation / blocked access where applicable

Comprehensive over selective. The point of an independent reference is to
catch causal links the CEE+ outputs may have missed.

5. **Effect choice** — most specific label that applies. `threatens` only when
   nothing else fits and the danger is direct + unavoidable.

## What I deliberately do NOT do

- Recommend actions (this is Graph B equivalent, not recommendations).
- Use presumed / inferred entities by default. The default policy is **only
  directly observable entities** appear in the GT graph. Off-frame fires,
  presumed occupants inside structures, unseen drivers, and other scene-
  implied entities are NOT included unless the user explicitly toggles this
  on for a given candidate.
- Add scene_summary, threats block, or any non-graph fields.

### When inferred IS allowed (opt-in)

If the user enables the "Allow inferred entities" toggle in the GT verifier
or asks for it during generation, I may add entities with `"inferred": true`
following the protocol's example patterns (e.g., `presumed_person_1` inside
a fully-involved structure fire with no visible occupants). The default
remains off.

## Output reproducibility

I document each candidate's `source` field as `"claude-code (in-terminal generation)"`.
The terminal session timestamp is implicit in the file's mtime. If the user
wants explicit versioning, I can add a `generation_date` field.
